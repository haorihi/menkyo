# 設計書：新・道路標識問題システム

このドキュメントは、運転免許試験アプリケーション向けに、高品質な200問の道路標識問題セットを構築するためのアーキテクチャと実施戦略をまとめたものです。

## 1. データ構造 (signs_new.json)

新スキーマは、メタデータの追跡、堅牢性、および1つの標識に対して複数の設問を紐付けられるように設計されています。

```json
[
  {
    "sign_id": "330", // 法令上の公式番号 (例: 330 は 一時停止)
    "metadata": {
      "name_jp": "一時停止",
      "name_en": "Stop",
      "category": "規制標識", // 規制, 警戒, 指示, 補助
      "official_code": "330"
    },
    "image": {
      "filename": "sign_330.png",
      "source_url": "https://commons.wikimedia.org/wiki/File:Japan_road_sign_330.svg"
    },
    "problems": [
      {
        "id": "q_330_01",
        "type": "truefalse",
        "question": "この標識がある場所では、徐行ではなく一時停止しなければならない。",
        "answer": true,
        "explanation": "「止まれ」は一時停止を義務付ける規制標識です。徐行（すぐに止まれる速度）では不十分で、タイヤを完全に止める必要があります。",
        "hikkake_point": "徐行との混同",
        "difficulty": 1
      },
      {

        "hikkake_point": "設置場所の限定",
        "difficulty": 2
      }

  }

```

## 2. 画像取得戦略

正確性と法的遵守を確実にするため、**ウィキメディア・コモンズ**のSVGファイルをソースとして使用します。

### 取得フロー:
1.  **マッピング**: 「公式標識番号」と「ウィキメディアのファイル名」の対応表を作成します（例: `330` -> `Japanese_road_sign_330.svg`）。
2.  **自動化スクリプト**: Pythonスクリプトを使用して以下の処理を行います。
    -   ウィキメディアAPIからSVGを取得。
    -   モバイル/Web環境での互換性のため、SVGをPNGに変換（必要に応じて）。
    -   ファイルを `data/signs_new/` に整理。
3.  **目視確認**: 各画像が現行の日本の道路交通法に基づいた正しいデザインであるかを確認します。

## 3. 問題作成戦略 (目標: 200問)

**約100種類のユニークな標識**に対し、それぞれ**2問ずつ**作成し、計200問を目指します。

### カテゴリ内訳:
| カテゴリ | 標識数 (目安) | 目標設問数 |
| :--- | :--- | :--- |
| **規制標識** | 45 | 90 |
| **警戒標識** | 35 | 70 |
| **指示標識** | 10 | 20 |
| **補助標識** | 10 | 20 |
| **合計** | **100** | **200** |

### 設問パターン:
-   **定義問題**: 標識の基本的な意味を問う。
-   **ひっかけ (応用)**: 適用範囲や例外などを問う（例: 「歩行者には適用されるか」「24時間適用か」）。
-   **比較問題**: 形や色が似ている標識との違いを問う（例: 駐車禁止 vs 駐停車禁止）。

## 4. 実施タイムライン

1.  **ステップ 1**: 100種類の標識リストとそのマッピングを確定。
2.  **ステップ 2**: 画像取得スクリプトの実行。
3.  **ステップ 3**: 200問のJSONデータ執筆。
4.  **ステップ 4**: 最終的な法規監査とバリデーション。
